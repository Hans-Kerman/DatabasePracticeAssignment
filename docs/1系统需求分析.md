## 一、 系统介绍 (System Overview)

图书管理系统是一款专为图书馆、学校图书室等场景设计的信息化管理工具，旨在帮助管理员高效管理馆藏图书、读者信息以及复杂的借阅业务。随着图书馆馆藏规模的扩大与读者服务需求的提升，传统的手工登记模式效率低下、数据易丢失，难以满足现代图书馆对管理精细化与服务便捷化的需求。

**核心目标与特色：**

1. **书目与实体单册分离（SPU与SKU思想）**：实现精确到每一本物理书的追踪。
2. **复合分类与读者评价**：完善检索与交流生态。
3. **标准化违约处理与缺书预约机制**：规范借阅秩序。
4. **全链路集成**：覆盖分类、单册管护、借阅流转、违约处理、库存统计与查询。

## 二、 系统模块划分 (System Modules)

为了方便前端组件划分和后端微服务/Controller拆分，系统结构规划如下：

### 2.1 系统模块树 (模块 -> 子功能)
- **1. 图书书目与分类管理**
  - 复合分类维护
  - 书目元数据建档
- **2. 馆藏单册物理管理**
  - 单本入库登记
  - 物理状态维护
- **3. 读者信息管理**
  - 读者开户
  - 读者信息维护
- **4. 借阅与归还管理**
  - 借阅办理
  - 归还办理
- **5. 逾期违约处理**
  - 罚单生成
  - 违约清缴
- **6. 预约与评价管理**
  - 缺书排队预约
  - 读后评价交流
- **7. 查询统计**
  - 馆藏与借阅查询
  - 业务报表统计
- **8. 系统维护**
  - 用户管理
  - 基础设置

### 2.2 模块功能详述 (业务逻辑提炼)
- **图书书目与分类管理**：
  - **分类**：支持将一本书归入多个交叉学科分类（多对多关系），便于多维检索。
  - **书目**：录入ISBN、书名、作者、出版社等抽象基础信息（不涉及物理库存）。
- **馆藏单册物理管理**：
  - **入库**：在对应“书目”下登记具体的实体副本，生成分配唯一 `id`。
  - **状态维护**：自动流转单册状态（**在馆 / 已借出 / 破损 / 遗失**）。
- **借阅与归还管理**：
  - **借阅**：扫读者证+单册id，校验权限与状态（必须为“在馆”），完成后更新状态为“已借出”。
  - **归还**：录入id，更新状态为“在馆”，记录实际归还时间。
- **逾期违约处理**：
  - 比对实际归还时间与应还时间。若逾期，按费率计算滞纳金，生成强依赖于该次借阅记录的违约罚单。未缴清罚单的读者将受到借阅限制。
- **预约与评价管理**：
  - **缺书预约**：当某书目下所有物理单册均为“已借出”时，发起排队预约。单册归还后，**按时间顺位自动通知**队列首位读者。

## 三、 客户端功能需求 (Client Requirements)

### 3.1 读者端 (Reader Client)

1. **注册登录**：学号/工号验证身份。
2. **复合检索与查询**：多级分类、ISBN、书名条件检索；可查看单册在馆状态；可浏览书评。
3. **借阅与预约**：对无可用单册的图书提交排队预约申请。
4. **个人中心**：查看当前借图书、历史借阅轨迹、排队中预约记录、未缴清罚单；修改联系信息；发表书评。

### 3.2 管理员端 (Admin Client)

1. **登录授权**：工号登录，基于角色分配后台权限。
2. **基础架构管理**：维护多级分类字典，录入图书元数据。
3. **读者与违约管理**：管理读者档案，办证/挂失，人工核销违约罚金。
4. **流转业务办理**：处理单册借阅、归还、续借；登记残损单册并作废。
5. **统计决策**：导出藏书结构报表、读者活跃度报表、罚金明细表。

## 四、 核心数据流逻辑 (Data Flow Logic)

*(注：此部分将原图表转化为数据流向描述，非常适合用于梳理后端 Service 层的互相调用关系)*

### 1. 顶层交互 (Level 0 / Level 1)
- **读者** <--> `[P3借阅归还管理]`, `[P4缺书预约管理]`, `[P5违约与评价管理]`
- **管理员** <--> `[P1基础信息管理]`, `[P2单本库存管理]`

### 2. 核心底层业务流转 (Level 2 DFD 转化为逻辑)

**A. 基础信息与单册维护**
- `输入读者信息` -> **[P1.1 读者开户]** -> 写入 `[D1读者信息表]`
- `输入书目信息` -> **[P1.2 书目元数据录入]** -> 写入 `[D2书目信息表]`
- `新书入库表` -> **[P2.1 单本录入登记]** -> 写入 `[D6馆藏单册表]`
- `单册报废表` -> **[P2.2 单册破损注销]** -> 更新 `[D6馆藏单册表]`状态

**B. 借还与违约流转 (极度重要：事务核心)**
- **借阅流转**：
    1. 接收 `借阅申请` -> **[P3.1 校验读者资格]** (查询 `D1读者状态`)。
    2. 资格通过 -> **[P3.2 登记借出]** -> 生成借出记录写入 `[D3借阅记录表]`，同时更新 `[D6馆藏单册表]` 状态为"已借出"。
- **归还与逾期流转**：
    1. 接收 `归还实体书` -> **[P3.3 登记归还]** -> 更新 `D3借阅表` (更新归还时间)，更新 `D6单册表` (状态改在馆)。
    2. 若 `实际归还时间 > 应还时间` -> 抛出超时数据至 **[P5.1 核算逾期违约金]** -> 写入 `[D8逾期罚单表]`。
    3. 读者缴纳罚金 -> **[P5.2 违约清缴]** -> 更新 `[D8逾期罚单表]` 缴费状态。

**C. 预约流转**
- 读者提交缺书单 -> **[P4.1 登记预约]** -> 写入 `[D7预约排队表]`。
- 接收来自P3的单册归还信号 -> **[P4.2 轮询到货分配]** -> 查询 `D7排队表` 提取首位读者 -> 更新预约状态 -> 发送取书通知。

## 五、 数据库字典设计 (Database Schema Dictionary)

*(注：这是为编写建表 SQL 准备的最核心内容，已经提取了主外键映射)*

### 5.1 核心数据表 (Data Stores)

| 存储编号 | 表名(建议英文名) | 数据结构 (字段组成) | 备注说明 |
| :--- | :--- | :--- | :--- |
| **D1** | 读者信息表 `user` / `reader` | **读者编号(PK)**, 姓名, 学号/工号, 联系方式, 账户状态 | 核心身份信息 |
| **D2** | 图书书目表 `book` | **图书编号(PK)**, ISBN(Unique), 书名, 作者, 出版社, 出版时间 | 仅记录抽象元数据(SPU) |
| **D3** | 借阅记录表 `lend_record` | **借阅编号(PK)**, **读者编号(FK)**, **单册id(FK)**, 借出时间, 应还时间, 实际归还时间 | 核心业务表，记录流转闭环 |
| **D4** | 分类字典表 `resource_category`| **分类编号(PK)**, 分类名称, 分类描述 | 独立存在的类别字典 |
| **D5** | 书目-分类映射表 `book_category`| **图书编号(PK/FK)** + **分类编号(PK/FK)** | 中间表，实现图书与分类的**多对多(M:N)**映射 |
| **D6** | 馆藏单册表 `book_copy` | **实体书编号(PK)**, **图书编号(FK)**, 书架位置, 当前物理状态(在馆/已借/遗失/破损) | 体现 1:N 关系(SKU)，直接反映库存 |
| **D7** | 预约排队表 `reservation` | **预约编号(PK)**, **读者编号(FK)**, **图书编号(FK)**, 提交时间, 预约状态 | 记录缺书时的请求序列 |
| **D8** | 逾期罚单表 `penalty` | **罚单编号(PK)**, **读者编号(FK)**, **借阅编号(FK)**, 产生金额, 缴清状态 | 强依赖于特定借阅记录的衍生实体 |
| **D9** | 图书评价表 `review` | **评价编号(PK)**, **读者编号(FK)**, **图书编号(FK)**, 评分(星级), 内容正文, 提交时间 | 读者间的交流信息固化存储 |

### 5.2 重要数据流定义表 (Data Flows)

*为API接口的 Request Body 设计提供参考：*

1. **单册入库单 (每月~300次)**：`实体书编号(SKU) + 图书ISBN(外键定位SPU) + 录入时间 + 初始保存状态`
2. **分类绑定信息 (每月~500次)**：`图书编号 + 分类编号`（用于维护D5表）
3. **违约金清缴凭单 (每月~100次)**：`罚单编号 + 读者编号 + 借阅记录编号(FK) + 缴纳金额 + 销账时间`
4. **图书评价单 (每月~600次)**：`评价编号 + 读者编号 + 图书编号 + 评价星级 + 评价正文 + 提交时间`

### 5.3 核心数据处理过程说明 (Stored Procedures / Triggers / Transactions)

这些过程描述了必须在数据库层（事务控制）或后端逻辑层严格实现的逻辑：

**【P3.1 单册借还办理】**
- **输入**: 读者信息(D1)、单册条码(D6)
- **输出**: 借阅记录表(D3)、馆藏单册表(D6)、逾期罚单表(D8)
- **事务逻辑处理**:
    1. 扫码核验读者资格。
    2. 若办理借出：向 D3 写入一条新记录，**同时**修改 D6 对应单册的状态为“已借出”。
    3. 若办理归还：更新 D3 的实际归还时间。**判断**若实际归还时间 > 应还时间，触发违约机制生成数据写入 D8。最后恢复 D6 单册状态为“在馆”。

**【P3.2 排队预约轮转】**
- **输入**: 归还单册信息(D6)、预约排队表(D7)
- **输出**: 预约排队表状态更新(D7)
- **事务/事件逻辑处理**:
    1. 当某书目的实体单册被归还入库时，触发检索。
    2. 查询 D7 是否存在该图书编号的待处理预约。
    3. 若存在，则按时间先后顺序，**锁定排在首位的读者的预约状态**，并通过系统向其发送取书通知。
